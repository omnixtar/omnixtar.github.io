## 一、渐进移植：让焊死的代码先长出一只“符式触手”

程序员不必放弃现有开发习惯。你依然可以写 Python、Java、C++，依然可以用变量焊死数据结构，依然可以跑传统 CI/CD 流程。

但 Omni*Web 提供了一个“旁路插槽”：  
**用 Phoscript 词元封装关键算法模块，作为传统系统的一个“外部大脑”。**

例如，你有一个 Java 微服务负责经纬度距离计算。不必重写整个服务，只需：  
1. 将距离计算逻辑提取为 Phoscript 词元 `geo-distance`；  
2. 通过 Phoscript 的 FFI（外部函数接口）从 Java 调用这个词元；  
3. 词元返回结果，Java 继续用变量接收。

**这一步，传统代码没有损失任何功能，却获得了三个新属性：**
- **可组合性**：`geo-distance` 从此可以被地图、物流、游戏等任何系统调用；  
- **可验证性**：词元经过逆图灵测试，结果精确可复现；  
- **可归属性**：Omnihash 绑定作者，调用即计费。

程序员不必“切换范式”，只需要在关键节点“插入符式元件”。传统系统像一艘旧船，逐渐把锈蚀的钢板换成新合金——从一块钢板开始，边航行边改装。

## 二、脱壳锐变：COBOL / FORTRAN 的“木乃伊复活术”

COBOL 和 FORTRAN 是计算机史的活化石。全球仍有数十亿行 COBOL 代码在银行、保险、政府核心系统里运行，但它们被锁在封闭的机房里，像被福尔马林泡着的标本，没人敢动，也没人懂。

Omni*Web 的移植策略面对它们时，展现出了独特的“考古学式温柔”：

1. **非侵入式封装**：为 COBOL 的子程序（SUBROUTINE）或 FORTRAN 的函数（FUNCTION）生成一个 Phoscript 词元外壳。词元内部通过系统调用（如 `system()` 或 C 桥接）运行原有编译好的二进制文件。  
   - 此时，COBOL 程序依然是那个 COBOL 程序，只是外面套了一层符式“翻译官”。

2. **数据流接管**：将 COBOL 的变量数据（如 `WS-AMOUNT PIC 9(8)V99`）映射到栈上的浮点数。COBOL 处理完毕后，结果被放回栈，而非写回变量。  
   - **关键点**：COBOL 内部的变量依然焊死，但**输入输出已经脱离了变量名的束缚**，变成了栈上的自由数据。

3. **逐步剥离**：随着符式词元库的增长，越来越多功能被 Phoscript 重写。当某个 COBOL 模块的所有子功能都有了对应的符式词元时，这个模块可以被整体替换——**像蛇蜕下旧皮一样，旧 COBOL 代码只是被标记为“只读历史”，不再被执行。**

这个过程可以在几十年的跨度上完成。银行不必停产，核心账务系统依然跑着 COBOL，但它的“API 层”已经变成了符式词元，可以被全世界的 Phoscript 应用自由组合调用。  
**化石活过来了——不再是一具被博物馆保护的尸体，而是一只被符式重新注入灵魂的电子生物。**
